# 跨源模式检测方法论

> 本文档详细描述7种跨源模式的检测算法、验证标准和证据链构建方法。
> 由主CC在Phase 3读取参考。

---

## 方法论基础

### 为什么需要跨源检测？

单源分析的局限性：

| 数据源 | 能看到 | 看不到 |
|--------|--------|--------|
| IM群聊 | 讨论了什么 | 讨论是否变成了行动 |
| 妙记 | 会上决定了什么 | 决定是否被执行 |
| 任务 | 什么逾期了 | 为什么逾期（沟通断点？瓶颈人？） |
| 文档 | 谁写了什么 | 什么该写但没人写 |
| 日历 | 开了多少会 | 会议是否产出了价值 |

跨源关联让我们看到**因果链**：消息 → 决策 → 任务 → 完成（或断裂）。

### 检测三原则

1. **双源验证**
   - 任何模式必须有≥2个独立数据源的证据
   - 单源异常标记为"信号"（Signal），不升级为"模式"（Pattern）
   - 信号记录在findings.md的附录中，供人工判断

2. **时序因果假设**
   - A发生在B之前 → 可以假设A可能影响B
   - 时间窗口限制：默认7天内（可按项目调整）
   - 始终标注"相关但未必因果"——让读者自行判断

3. **统计显著性阈值**
   - 频率变化>30%才标记为异常
   - 基于前N天的滑动平均作为基线
   - 消除周末/节假日/特殊事件的干扰

## P1: 决策漂移详细检测

### 输入
- Minutes数据：每次会议的决策列表
- IM数据：群聊中包含"决定/确认/同意/改为/重新/推翻"等决策词的消息

### 算法

```
Step 1: 决策提取
  从Minutes和IM中提取所有决策语句
  标注：时间 + 来源 + 内容 + 参与者

Step 2: 主题聚类
  按关键词相似度将决策分组
  同一主题的决策归入同一簇

Step 3: 一致性检查
  对每个决策簇：
    IF 簇内决策数 ≥ 2 AND 结论不一致
    THEN 标记 Decision Drift
    严重度 = 不一致次数 × 时间跨度

Step 4: 影响评估
  每次翻转的代价 = 翻转后到下一次翻转的时间（执行了又推翻的浪费）
```

### 验证标准
- 必须有具体的决策对比（不能是模糊的"讨论了这个话题"）
- 结论不一致需要明确说明差异点
- 排除：正常的渐进式细化（"先做A，后来发现B更好"是进步不是漂移）

## P2: 沉默瓶颈详细检测

### 输入
- IM数据：@关系矩阵 + 消息时间戳
- Task数据：任务负责人 + 任务启动时间

### 算法

```
Step 1: 枢纽识别
  计算每人被@频率
  被@频率 > 团队均值 × 3 → 标记为潜在瓶颈

Step 2: 响应延迟计算
  对每个潜在瓶颈人：
    平均响应时间 = mean(回复时间 - 被@时间)
    仅计算工作时间内的延迟

Step 3: 下游影响关联
  检查：瓶颈人回复后 → 相关任务状态变更的时间差
  IF 回复延迟与任务启动延迟呈正相关
  THEN 确认为 Silent Bottleneck

Step 4: 影响量化
  周均等待时间 = 被@次数/周 × 平均响应延迟
```

### 验证标准
- 被@频率必须统计显著（>3倍均值，排除仅1-2次的偶发@）
- 响应延迟必须基于>10次@的样本
- 下游影响需要至少3个关联实例

## P3: 知识蒸发详细检测

### 输入
- IM数据：群聊消息（含技术讨论识别）
- Doc数据：文档创建时间线

### 算法

```
Step 1: 技术讨论识别
  在IM中检测高密度讨论段：
    连续>15条消息 + 包含技术关键词 + 时间跨度<2小时
  标记为"重要讨论"

Step 2: 文档化检查
  对每个重要讨论：
    检查讨论后7天内是否有包含相关关键词的文档创建
    IF 无相关文档 → 标记为"未文档化讨论"

Step 3: 重复检测
  检查"未文档化讨论"的主题是否在后续IM中被重新提问
  IF 相同话题被第二次讨论 → 确认Knowledge Evaporation

Step 4: 影响评估
  知识损失成本 = 重复讨论次数 × 平均讨论参与人数 × 平均讨论时长
```

### 验证标准
- "重要讨论"需要>15条消息（排除日常闲聊）
- "相关文档"的关键词匹配需要>60%的主题词重叠
- "重复提问"必须由不同人发起（排除同一人追问）

## P4: 危机前静默详细检测

### 输入
- IM数据：每日消息量时间序列
- Task数据：任务逾期事件
- Calendar数据：紧急/临时会议

### 算法

```
Step 1: 基线计算
  计算前14天的消息量滑动平均（排除周末）
  标准差 = 日消息量的standard deviation

Step 2: 异常检测
  IF 连续2天消息量 < 基线 - 2×标准差 (即下降>40%)
  THEN 标记为"异常静默期"

Step 3: 后果关联
  检查静默期后5-10天内是否出现：
    - 任务逾期（Task）
    - 临时紧急会议（Calendar，非周期性且临时添加）
    - 升级关键词（IM中出现"紧急/blocked/延期/出问题/事故"）
  IF 出现上述任一事件 → 确认 Pre-Crisis Silence

Step 4: 预警价值评估
  潜在提前发现天数 = 静默开始日 到 问题爆发日 的间隔
```

### 验证标准
- 静默必须是异常的（>2个标准差），不是周末/假期
- 后果事件必须在时间窗口内（5-10天）
- 排除已知原因的静默（如团队外出/公司假期）

## P5: 会议通胀详细检测

### 输入
- Calendar数据：会议频率和时长趋势
- Task数据：任务完成率趋势

### 算法

```
Step 1: 会议趋势
  按周统计：会议次数、总时长、平均时长、参会人数

Step 2: 执行趋势
  按周统计：任务完成数、完成率

Step 3: 相关性分析
  计算会议频率与任务完成率的相关系数
  IF 相关系数 < -0.3 (负相关) OR 会议↑ AND 完成率持平
  THEN 标记 Meeting Inflation

Step 4: 效率指标
  会议占工作时间比 = 周会议总时长 / (参与者平均数 × 40h)
  健康区间：<20%
  预警区间：20-35%
  危险区间：>35%
```

## P6: 承诺衰减详细检测

### 输入
- Minutes数据：Action Items列表（含负责人+截止日）
- Task数据：任务创建和完成记录

### 算法

```
Step 1: 承诺提取
  从Minutes中提取所有Action Items
  标注：承诺内容 + 负责人 + 截止日 + 来源会议

Step 2: 任务匹配
  对每个承诺：
    搜索Task中是否有匹配的任务（关键词+负责人+时间窗口）
    匹配 → 记录任务ID
    未匹配 → 标记为"未任务化承诺"

Step 3: 完成追踪
  对已匹配的任务：
    检查是否在截止日前完成
    完成 → "已闭环"
    未完成或逾期 → "承诺未兑现"

Step 4: 漏斗计算
  承诺数 → 任务化数(任务化率) → 完成数(完成率) → 按时完成数(按时率)
  整体闭环率 = 按时完成数 / 总承诺数

Step 5: 按人分析
  每人的闭环率排名
  识别系统性低闭环率（全团队低）vs 个人差异
```

## P7: 信息孤岛详细检测

### 输入
- IM数据：多个群的消息（含关键词和结论）

### 算法

```
Step 1: 跨群话题匹配
  对每个群提取讨论主题（关键词聚类）
  跨群匹配：IF 群A和群B在相近时间段讨论了相同主题

Step 2: 结论对比
  提取每个群对该主题的结论或决定
  IF 结论不一致 → 标记为潜在信息孤岛

Step 3: 桥接检查
  检查两群是否有共同成员参与了该话题的讨论
  IF 有共同成员但未转发信息 → "桥接失效"
  IF 无共同成员 → "结构性孤岛"

Step 4: 影响评估
  从信息不一致发现日起，到两群认知对齐日（如果有）的时间差
  该时间内基于错误假设的工作量 = 浪费
```

## 模式优先级排序

当多个模式被检测到时，按以下维度排序（加权评分）：

| 维度 | 权重 | 说明 |
|------|------|------|
| 影响范围 | 30% | 影响多少人/多少任务 |
| 时间成本 | 30% | 估算浪费的人·天 |
| 可改善性 | 25% | 改进建议的可执行度 |
| 置信度 | 15% | 证据链的强度 |

评分 = Σ(维度分数 × 权重)，按总分降序排列。
