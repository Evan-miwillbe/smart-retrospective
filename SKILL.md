# 智能项目复盘 v1.0

> 传统复盘靠记忆，智能复盘靠证据。人看到的是单棵树，AI看到的是整片森林。

---

## 核心理念

```
传统复盘                          智能复盘
┌─────────────┐                 ┌──────────────────────────────┐
│ 人工回忆     │                 │ 数据重建                       │
│ → 选择性记忆 │                 │ → 跨5源交叉验证                │
│ → 归因偏差   │                 │ → 证据链完整可追溯             │
│ → 后见之明   │                 │ → 发现人眼不可见的跨源模式      │
└─────────────┘                 └──────────────────────────────┘
```

**单源分析 vs 跨源分析**：
- 看群聊：只知道"讨论了很多"
- 看任务：只知道"逾期了几个"
- **跨源**：发现"讨论量突然下降3天后，任务开始逾期"——这是隐藏的因果信号

## Phase 1: 多Agent并行数据采集

### 采集架构

```
主CC (调度者)
  ├── spawn → IM-Collector      [lark-im]
  ├── spawn → Minutes-Collector [lark-minutes]
  ├── spawn → Task-Collector    [lark-task]
  ├── spawn → Doc-Collector     [lark-doc + lark-wiki]
  └── spawn → Calendar-Collector[lark-calendar]
       ↓ 各自写入 00_collection/{source}/
  主CC ← 验证产出（Glob + Read确认非空）
```

### Agent Spawn模板

每个采集Agent的spawn prompt必须包含：
1. **范围**：具体的群ID/时间范围/token（不写模糊指令）
2. **输出路径**：绝对路径，写到哪个文件
3. **输出格式**：结构化markdown，包含时间戳+来源标注
4. **字数硬限**：每Agent最多输出5000字（防止token爆炸）
5. **摘要要求**：采集完成后必须写summary.md（关键统计+初步发现）

### 各Agent采集内容

| Agent | 数据源 | 采集字段 | 输出 |
|-------|--------|---------|------|
| IM-Collector | 群消息 | 时间、发送者、内容摘要、@关系、消息类型 | im/messages.md + im/summary.md |
| Minutes-Collector | 妙记 | 会议主题、参与者、关键讨论、决策、action items | minutes/transcripts.md + minutes/summary.md |
| Task-Collector | 任务 | 标题、负责人、创建/截止/完成时间、状态变更 | tasks/timeline.md + tasks/summary.md |
| Doc-Collector | 文档 | 标题、创建者、创建/编辑时间、协作者、摘要 | docs/inventory.md + docs/summary.md |
| Calendar-Collector | 日历 | 会议标题、时间、参与者、时长、是否取消/改期 | calendar/events.md + calendar/summary.md |

### 采集验证

所有Agent完成后，主CC逐个验证：
- [ ] 文件存在（Glob）
- [ ] 文件非空且格式正确（Read前20行）
- [ ] summary.md包含关键统计数字
- 任一Agent失败 → 重试1次；仍失败 → 标记该数据源不可用，继续

## Phase 2: 指标引擎

主CC读取所有采集数据，计算四维指标体系：

### 2.1 沟通维度

| 指标 | 计算方式 | 意义 |
|------|---------|------|
| 消息频率曲线 | 按日聚合消息数 | 项目活跃度趋势 |
| @网络密度 | 谁@谁的频率矩阵 | 信息流向与依赖关系 |
| 响应时间中位数 | 被@到回复的时间差 | 沟通效率 |
| 话题集中度 | 高频关键词占比 | 讨论是否聚焦 |

### 2.2 执行维度

| 指标 | 计算方式 | 意义 |
|------|---------|------|
| 任务创建→完成率 | 已完成/总创建 | 整体执行力 |
| 平均任务周期 | 创建到完成的天数 | 交付速度 |
| 逾期率趋势 | 按周统计逾期比例 | 是否越来越赶 |
| 任务堆积量 | 未完成任务的累积曲线 | 负荷是否失控 |

### 2.3 会议维度

| 指标 | 计算方式 | 意义 |
|------|---------|------|
| 会议频率 | 按周统计会议次数 | 协调开销 |
| 平均时长趋势 | 会议时长是否在膨胀 | 效率信号 |
| 决策密度 | 每次会议产出的决策数 | 会议产出率 |
| 取消/改期率 | 取消或改期的比例 | 优先级冲突信号 |

### 2.4 知识维度

| 指标 | 计算方式 | 意义 |
|------|---------|------|
| 文档创建频率 | 按周统计新文档数 | 知识沉淀速度 |
| 协作密度 | 每文档的编辑者数 | 知识共享程度 |
| 文档存活率 | 创建后30天内被编辑的比例 | 文档是否活文档 |

输出：`01_metrics/dashboard.md`（四维指标汇总表+趋势描述）

## Phase 3: 跨源模式检测（核心创新）

> 单源数据只能看到现象，跨源关联才能发现因果。

### 模式检测原则

1. **双源验证**：每个模式必须有≥2个数据源的交叉证据，单源信号仅标记"待验证"
2. **时序因果**：A→B要求A在时间上先于B，且间隔合理（不能相隔3个月还算因果）
3. **统计显著**：频率变化>30%才标记为异常（消除日常波动噪声）
4. **证据链完整**：每个发现必须附带可追溯的[源数据引用]

### 7种核心模式

#### P1: 决策漂移 (Decision Drift)

**定义**：同一议题在多次会议/讨论中出现且结论不一致——团队在原地打转。

**检测逻辑**：
```
Minutes中提取决策 → 按主题关键词聚类 → 
检查同主题是否在后续Minutes/IM中被推翻或重新讨论 →
IF 同主题决策出现≥3次且结论不同 → 标记Decision Drift
```

**证据链格式**：
```
[妙记A, 4月3日] 决定采用方案X
  → [群聊, 4月5日] "方案X成本太高，需要重新评估"
  → [妙记B, 4月8日] 改为方案Y
  → [群聊, 4月12日] "方案Y也有问题..."
影响：累计浪费≈2周决策时间
```

---

#### P2: 沉默瓶颈 (Silent Bottleneck)

**定义**：某人是多条信息流的交汇节点，其响应速度直接影响整体velocity——但没人意识到。

**检测逻辑**：
```
IM中统计每人被@频率 → 排名top 10% → 
计算其平均响应时间 → 
关联：该人回复后 → 下游任务启动的时间差 →
IF 响应延迟与下游启动延迟强正相关 → 标记Silent Bottleneck
```

**证据链格式**：
```
[张三] 跨3个群被@频率 = 12次/天（团队均值3次/天）
  平均响应时间：6.5小时
  关联效应：张三回复后，下游任务平均4小时内启动
  瓶颈影响：每周因等待张三回复累计延迟≈18小时
```

---

#### P3: 知识蒸发 (Knowledge Evaporation)

**定义**：重要技术讨论发生在群聊中但未被文档化，导致知识随消息沉没而丢失。

**检测逻辑**：
```
IM中识别高密度技术讨论（连续>20条消息+技术关键词） →
检查讨论后7天内是否有相关Doc/Wiki创建 →
检查同一话题是否在后续IM中被重新提问 →
IF 无文档创建 AND 话题被重复提问 → 标记Knowledge Evaporation
```

**证据链格式**：
```
[4月5日] 群内讨论数据库选型（38条消息，4人参与，持续2小时）
  → [未创建文档]
  → [4月22日] 新同事问"我们为什么选了PostgreSQL而不是MongoDB？"
  → 只有2人能回答，且回忆不一致
知识损失：选型决策理由未沉淀，onboarding成本增加
```

---

#### P4: 危机前静默 (Pre-Crisis Silence)

**定义**：项目群沟通频率的异常下降，往往预示着即将到来的问题（逾期、事故、冲突）。

**检测逻辑**：
```
计算IM消息频率的滑动平均 →
检测频率下降>40%的时段 →
检查该时段后5-10天内是否出现：
  - Task逾期
  - 紧急会议（Calendar中临时添加的会议）
  - 升级讨论（IM中出现"紧急/blocked/延期"等关键词）
IF 静默后出现上述事件 → 标记Pre-Crisis Silence
```

**证据链格式**：
```
[4月10-13日] 群消息量从日均45条降至日均12条（-73%）
  → [4月16日] 3个关键任务标记逾期
  → [4月17日] 紧急会议（临时安排，6人参加）
回溯分析：静默期间关键开发人员在处理未上报的技术障碍
预警价值：如果在4月11日检测到异常静默并主动询问，可提前5天发现问题
```

---

#### P5: 会议通胀 (Meeting Inflation)

**定义**：会议频率持续增加但任务完成率未同步提升——会议正在从工具变成负担。

**检测逻辑**：
```
Calendar统计周会议数趋势 →
Task统计周任务完成数趋势 →
计算相关性：
IF 会议数↑ AND 完成率持平或↓ → 标记Meeting Inflation
附加检测：会议时长是否也在膨胀
```

**证据链格式**：
```
[3月] 周均会议4次，任务完成率78%
[4月上旬] 周均会议7次（+75%），任务完成率71%（-7%）
[4月下旬] 周均会议9次（+125%），任务完成率65%（-13%）
会议时间占比：从20%升至45%（超过健康阈值30%）
建议：至少3个周期性会议可以改为异步文档同步
```

---

#### P6: 承诺衰减 (Commitment Decay)

**定义**：会议中的承诺在"说→记→做"的链条中逐级衰减。

**检测逻辑**：
```
Minutes中提取所有action items（含负责人+截止日） →
检查每个item是否在Task系统中有对应任务 →
检查对应任务是否在截止日前完成 →
计算：承诺转化率 = 任务化率 × 完成率
```

**证据链格式**：
```
8次会议共提取42个承诺:
  → 28个创建了飞书任务（任务化率 67%）
  → 28个任务中19个按时完成（按时完成率 68%）
  → 整体闭环率：42→19 = 45%
最大衰减阶段："说了但没建任务"（33%丢失）
高衰减人员：李四(闭环率22%)、王五(闭环率30%)
```

---

#### P7: 信息孤岛 (Information Silo)

**定义**：不同群/团队在讨论相同话题但结论不同，且无信息桥接。

**检测逻辑**：
```
多群IM按关键词做话题匹配 →
检测同话题在不同群中的结论差异 →
检查两群是否有共同成员参与该话题讨论 →
IF 话题重叠 AND 结论不同 AND 无交叉讨论 → 标记Information Silo
```

**证据链格式**：
```
[前端群, 4月8日] "API接口将在5月1日上线，我们按这个排期"
[后端群, 4月10日] "API重构延期到5月15日，已通知产品"
  → 未通知前端群
  → 前端群直到4月25日才发现延期（浪费17天按错误假设开发）
桥接缺失：两群仅1个共同成员，但该成员未转发信息
```

## 铁律

1. **跨源才是模式** — 单源发现只是"信号"，两源交叉验证才升级为"模式"
2. **证据链不可断** — 每个发现必须可追溯到具体数据源（时间戳+来源标注）
3. **量化优于定性** — "沟通效率下降"不如"响应时间从2h增至8h"
4. **时序即因果假设** — A先于B才能假设A影响B，标注"相关≠因果"
5. **30%阈值** — 变化<30%视为正常波动，不标记为异常
6. **采集Agent字数硬限** — 每Agent≤5000字，超出即截断（防止token爆炸）
7. **不做归咎，只做归因** — 报告用"发现"不用"责任"，指向系统问题而非个人
8. **数据不足时降级** — 某源无数据不阻塞整体流程，标注后继续
